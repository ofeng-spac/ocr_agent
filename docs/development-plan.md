# 药品识别 Agent 开发文档

## 1. 文档目标

本文档用于定义项目下一阶段开发的统一基线。

后续开发默认以本文档为准，除非项目范围被明确调整。

当前项目的目标不是单纯把视频送进大模型并输出一个名字，而是把现有的视频识别能力升级为一个前后端一体的 Agent 系统，核心围绕以下五件事展开：

1. 视频药品识别
2. 说明书字段抽取
3. 基于说明书名称字段的标准名校验
4. 基于说明书内容的 RAG 检索与解释
5. 全流程可追踪的工作流编排

## 2. 项目定位

### 2.1 最终定位

系统定位为：

`药品视频识别与说明书校验 Agent 平台`

### 2.2 核心价值

系统不直接信任大模型最终输出的药品名称。

系统的核心逻辑是：

1. 先用 VLM 从视频中识别出原始药品名称线索
2. 再从说明书中抽取权威药品名称字段
3. 用说明书构建标准药品目录
4. 用标准目录对大模型结果做校验与归一
5. 只有匹配成功时才输出标准药品名
6. 匹配失败时输出 `review_required` 或 `unknown`

这套设计是项目的核心抗幻觉机制。

### 2.3 面试主线

这个项目不是通用 OCR。

它是一个面向药品包装识别场景的、受领域知识约束的多模态识别系统，结合了：

1. 视频抽帧
2. VLM 多帧识别
3. 说明书结构化字段抽取
4. 标准名归一
5. RAG 检索
6. LangGraph 工作流编排

## 4. 开发原则

### 4.1 第一原则

识别准确率和抗幻觉能力优先于“看起来很复杂的 Agent 壳子”。

### 4.2 范围原则

第一阶段只围绕当前 11 个类别开展。

暂时不做真实医院业务系统接入，不做 HIS/FHIR 对接，不做病人医嘱流转。

### 4.3 信任原则

最终标准药品名称必须同时满足两个条件：

1. 来自视频识别证据
2. 能被说明书名称字段校验通过

不能因为模型“看起来说得像”就直接当作最终结果。

### 4.4 工程原则

必须复用现有识别核心。

除非有明确缺陷，否则不重写 `src/server.py` 主流程。

## 5. 产品范围

### 5.1 本阶段包含

1. 样例视频选择或视频上传
2. 基于 VLM 的原始药品名识别
3. 基于说明书字段的标准名校验与归一
4. 说明书检索与解释
5. 审计日志记录
6. 评测结果展示
7. LangGraph 工作流执行轨迹展示

### 5.2 本阶段不包含

1. 完整医院“三查七对”系统
2. 真实病人、病区、医嘱、护士数据管理
3. 账号权限系统
4. 完整文档运营平台
5. 多智能体复杂协作

## 6. 核心功能设计

### 6.1 功能 A：视频药品识别

输入：

1. 用户上传视频，或
2. 用户选择样例视频

输出：

1. 原始识别药品名 `raw_name`
2. 证据文本 `evidence_text`
3. 不确定性等级 `uncertainty`
4. 推理耗时 `latency`
5. 抽帧预览结果

说明：

1. 这一层只做“识别”
2. 不负责最终标准名判定

### 6.2 功能 B：标准名校验与归一

这是整个系统最重要的功能。

输入：

1. 视频识别得到的 `raw_name`
2. 基于说明书抽取出来的标准药品目录

输出：

1. `canonical_name`
2. `verify_status`
3. `match_type`
4. 如果未通过，返回失败原因

状态定义：

1. `verified_exact`
2. `verified_alias`
3. `verified_brand`
4. `review_required`
5. `unknown`

匹配类型定义：

1. 标准通用名完全匹配
2. 商品名匹配
3. 别名匹配
4. 模糊匹配但证据充分
5. 未匹配

### 6.3 功能 C：说明书检索

当标准名校验通过后，系统允许继续对说明书内容做检索和回答。

第一阶段支持的问题类型：

1. 药品正式名称是什么
2. 规格是什么
3. 适应症是什么
4. 用法用量是什么
5. 注意事项有哪些
6. 禁忌有哪些

要求：

1. 回答必须带来源片段
2. 回答不得脱离检索文本自由发挥

### 6.4 功能 D：核验模式

用户可以输入一个期望药品名称 `expected_drug_name`。

系统将其与最终归一后的 `canonical_name` 比较，输出：

1. `match`
2. `mismatch`
3. `review_required`

### 6.5 功能 E：审计日志

所有请求都必须记录。

需要记录的最小信息包括：

1. 输入来源
2. 原始识别结果
3. 标准名校验结果
4. 检索问题与回答
5. 工作流执行状态
6. 时间戳

### 6.6 功能 F：评测看板

系统需要基于当前标注数据集提供离线评测能力。

至少展示：

1. 总体准确率
2. 各类别样本数
3. 典型错误案例
4. 平均耗时

## 7. 抗幻觉设计

### 7.1 基本原则

大模型输出的是“证据候选”，不是“最终真值”。

### 7.2 校验流程

1. VLM 输出 `raw_name`
2. 系统提取可用证据文本
3. 系统在标准药品目录中查找候选项
4. 系统判断是否能够归一到标准药名
5. 只有归一成功时才输出最终结果

### 7.3 拒答规则

遇到下列情况时，系统不能强行返回一个标准药名：

1. 在目录中找不到可靠候选项
2. 有多个候选项同时成立
3. 只存在弱模糊匹配
4. 文档缺失且证据不足

此时只能返回：

1. `review_required`
2. `unknown`

## 8. 数据模型设计

### 8.1 `drug_catalog.json`

这个文件是标准名校验的权威数据源。

示例：

```json
[
  {
    "drug_id": "drug_001",
    "canonical_name": "注射用头孢噻呋钠",
    "generic_name": "注射用头孢噻呋钠",
    "brand_names": [],
    "aliases": ["头孢噻呋钠"],
    "doc_paths": ["data/source_docs/text/头孢噻呋钠.txt"],
    "domain": "vet",
    "category": "powder_for_injection",
    "active": true
  }
]
```

必须包含的字段：

1. `drug_id`
2. `canonical_name`
3. `generic_name`
4. `brand_names`
5. `aliases`
6. `doc_paths`
7. `domain`
8. `category`
9. `active`

### 8.2 `leaflet_fields.jsonl`

这个文件保存从说明书中抽取出的结构化字段。

必须字段：

1. `drug_id`
2. `canonical_name`
3. `field_name`
4. `field_value`
5. `source_file`
6. `source_type`

第一阶段支持的字段类型：

1. `generic_name`
2. `brand_name`
3. `specification`
4. `indications`
5. `dosage`
6. `precautions`
7. `contraindications`
8. `appearance`

### 8.3 `eval_records.csv`

这个文件用于记录结构化评测结果。

建议字段：

1. `video_name`
2. `ground_truth`
3. `raw_name`
4. `canonical_name`
5. `verify_status`
6. `matched`
7. `latency_sec`
8. `config`

### 8.4 `audit_logs` 表

使用 SQLite 存储。

建议字段：

1. `trace_id`
2. `request_type`
3. `video_name`
4. `expected_drug_name`
5. `raw_name`
6. `canonical_name`
7. `verify_status`
8. `question`
9. `answer`
10. `latency_sec`
11. `created_at`

## 9. 文档处理设计

### 9.1 文档来源管理

当前零散的 PDF/TXT 文件必须迁移到项目内部统一管理。

目标目录：

1. `data/source_docs/raw/`
2. `data/source_docs/text/`
3. `data/structured/`

### 9.2 文档处理流程

1. 收集原始文档
2. PDF 转文本
3. 清洗 OCR 噪音和重复页眉页脚
4. 按章节切分
5. 提取结构化字段
6. 生成检索 chunk
7. 生成标准药品目录数据

### 9.3 重点章节

优先抽这些章节：

1. 药品名称
2. 规格
3. 适应症
4. 用法用量
5. 注意事项
6. 禁忌
7. 性状

## 10. 名称匹配规则

### 10.1 匹配优先级

匹配顺序如下：

1. 标准名完全匹配
2. 通用名完全匹配
3. 商品名完全匹配
4. 别名完全匹配
5. 标点标准化后的匹配
6. 高置信模糊匹配

### 10.2 标准化规则

匹配前需要做文本标准化：

1. 去首尾空格
2. 统一全角半角标点
3. 在必要时去掉 `®` 等装饰符号
4. 保留可能有医学意义的限定词
5. 原始文本必须保留在日志中

### 10.3 模糊匹配限制

模糊匹配只有在以下条件同时满足时才能生效：

1. 名称相似度足够高
2. 证据文本支持该候选项
3. 没有其他竞争候选项得分接近

否则必须返回 `review_required`。

## 11. RAG 设计

### 11.1 RAG 的定位

RAG 是辅助能力，不是最终标准名判定依据。

它主要负责：

1. 检索说明书内容
2. 回答受约束的问题
3. 提供可引用的证据片段

### 11.2 检索范围

RAG 不应该每次都全库盲搜。

标准流程是：

1. 先确认标准药品名
2. 再只检索该药对应说明书
3. 必要时再扩展到少量候选药品

### 11.3 Chunk 方案

优先按章节切分，再按长度细分。

每个 chunk 必须保留：

1. `drug_id`
2. `canonical_name`
3. `section`
4. `source_file`

### 11.4 向量数据库选型

第一阶段向量数据库选型：

`Qdrant`

理由：

1. 足够支持本地 Demo
2. 面试时易于说明
3. 运维成本低于更重的分布式方案

## 12. LangGraph 工作流设计

### 12.1 工作流目标

LangGraph 用于编排整个多步骤流程，并向前端暴露执行轨迹。

### 12.2 节点设计

计划节点如下：

1. `load_video`
2. `sample_frames`
3. `vision_recognize`
4. `extract_raw_fields`
5. `load_drug_catalog`
6. `verify_canonical_name`
7. `retrieve_leaflet_chunks`
8. `answer_leaflet_question`
9. `build_response`
10. `write_audit_log`

### 12.3 条件分支

1. 如果标准名校验失败，进入 `review_required`
2. 如果没有说明书，跳过检索并标记 `doc_unavailable`
3. 如果没有提问，跳过回答生成

### 12.4 工作流输出要求

每个 trace 至少展示：

1. 执行过的节点
2. 每个节点状态
3. 中间关键结果
4. 最终判定结果

### 13.3 API 设计

#### `POST /api/recognize`

用途：

输入视频，返回识别结果和标准名校验结果。

返回内容：

1. `raw_name`
2. `canonical_name`
3. `verify_status`
4. `evidence_text`
5. `latency`
6. `trace_id`

#### `POST /api/verify`

用途：

输入视频和期望药名，判断是否一致。

返回内容：

1. `expected_drug_name`
2. `canonical_name`
3. `verification_result`
4. `reason`
5. `trace_id`

#### `POST /api/rag/ask`

用途：

对已校验通过的药品做说明书问答。

返回内容：

1. `answer`
2. `source_chunks`
3. `source_files`
4. `trace_id`

#### `POST /api/kb/reindex`

用途：

重建结构化文档数据和向量索引。

#### `GET /api/audit_logs`

用途：

查询历史日志。

#### `GET /api/eval/summary`

用途：

返回当前模型配置下的评测汇总。

#### `GET /api/workflow/{trace_id}`

用途：

返回该次请求的工作流执行详情。

## 16. 验收标准

### 16.1 识别验收

1. 系统能跑通样例视频
2. 输出结构化识别结果
3. 不允许跳过标准名校验

### 16.2 校验验收

1. 原始大模型结果必须保留
2. 标准名校验过程必须明确可见
3. 未匹配场景必须返回拒答状态

### 16.3 RAG 验收

1. 回答必须带来源片段
2. 回答必须基于说明书文本
3. 缺失文档时必须明确提示

### 16.4 工作流验收

1. 每个请求都有 trace_id
2. 每个 trace 都能看见执行节点
3. 每个请求都有日志

### 16.5 页面验收

1. 核心操作都能在前端看见
2. 用户能知道结果为什么通过或不通过
