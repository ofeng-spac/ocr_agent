# 第二层检索升级前后效果与性能对比

## 1. 文档目的

本文档用于对比第二层检索在两种向量化方案下的差异：

1. 轻量哈希向量器版本
2. 真实中文 embedding 版本（`BAAI/bge-small-zh-v1.5`）

对比重点包括：

1. 检索质量
2. 语义问题效果
3. 平均延迟
4. P95 延迟
5. 工程取舍

## 2. 对比对象

### 2.1 升级前

- 向量方式：轻量哈希向量器
- 代码位置：`app/services/vectorizer.py`
- 特征：
  - 完全本地
  - 无模型依赖
  - 构建速度快
  - 语义表达能力有限

### 2.2 升级后

- 向量方式：真实中文 embedding
- 模型：`BAAI/bge-small-zh-v1.5`
- 代码位置：`app/services/embedding.py`
- 特征：
  - 本地模型推理
  - 真正的语义向量
  - 检索质量更高
  - 构建和查询耗时增加

## 3. 评测集

两次对比均基于同一套第二层检索评测集：

- 总样例数：`112`
- 字段问题：`92`
- 语义问题：`12`
- 拒答问题：`8`

评测脚本：

- `scripts/eval_rag.py`

评测集：

- `data/eval/rag_eval_cases.jsonl`

## 4. 质量指标对比

### 4.1 升级前（哈希向量器）

- 整体通过率：`99.11%`
- 字段问题通过率：`100.00%`
- 语义问题通过率：`91.67%`
- 拒答正确率：`100.00%`
- 平均延迟：`0.92 ms`
- P95 延迟：`1.11 ms`

### 4.2 升级后（真实 embedding）

- 整体通过率：`100.00%`
- 字段问题通过率：`100.00%`
- 语义问题通过率：`100.00%`
- 拒答正确率：`100.00%`
- 平均延迟：`5.01 ms`
- P95 延迟：`6.11 ms`

## 5. 关键结论

### 5.1 质量提升

真实 embedding 升级后，第二层最明显的变化是：

1. 整体通过率从 `99.11%` 提升到 `100.00%`
2. 语义问题通过率从 `91.67%` 提升到 `100.00%`

说明：

1. 字段问题本身已经比较稳定，升级前后差异不大
2. 升级主要改善的是开放式语义问题

### 5.2 性能代价

真实 embedding 升级后，第二层延迟明显上升：

1. 平均延迟从 `0.92 ms` 增加到 `5.01 ms`
2. P95 延迟从 `1.11 ms` 增加到 `6.11 ms`

说明：

1. 质量提升来自更强的语义表达能力
2. 性能代价来自模型向量化开销

### 5.3 工程判断

在当前数据规模和接口负载下，这个代价仍然可接受。

因为：

1. 第二层接口仍保持毫秒级响应
2. 语义问题质量提升是确定的
3. 这一提升非常适合用于项目展示和简历表述

## 6. 当前建议

### 6.1 如果目标是演示和面试

推荐保留真实 embedding 版本。

理由：

1. 语义问题质量更高
2. 指标更完整
3. 更符合“正式向量检索”表述

### 6.2 如果目标是极端低延迟

可以保留哈希向量器版本作为轻量回退方案。

理由：

1. 不依赖模型
2. 几乎零加载成本
3. 更适合非常轻量的本地验证场景

## 7. 适合写进简历的表述

推荐写法：

> 将第二层说明书检索从轻量哈希向量器升级为本地真实中文 embedding（`BAAI/bge-small-zh-v1.5`）+ Qdrant 检索，在自建 `112` 条评测集上将整体通过率从 `99.11%` 提升到 `100%`，语义类问题通过率从 `91.67%` 提升到 `100%`；同时平均检索延迟由 `0.92 ms` 增加到 `5.01 ms`，完成了效果与性能的量化权衡分析。
